# not-general-number-field-sieve

It's definitely not a GNFS.

## Lucky-number sparse-family search

This is a Rust backend for testing sparse integer families for lucky-number
membership. It started as a base-10 repdigit search, but the membership engine is
family-independent: each candidate is tracked by survivor rank through the lucky
sieve.

The default family is still decimal repdigits, matching OEIS A031882
(`A000959` lucky numbers intersect `A010785` repdigits). Additional sparse
families can be searched in the same sieve run:

- `repdigit`: decimal repdigits, OEIS A031882.
- `mersenne`: numbers of the form `2^k - 1`, OEIS A057613 intersection target.
- `mersenne-prime-exp`: `2^p - 1` for prime `p`, OEIS A057612 intersection target.
- `fibonacci`: Fibonacci candidates, OEIS A057589 intersection target.
- `lucas`: Lucas candidates, OEIS A306632 intersection target.
- `tetranacci`: tetranacci candidates, OEIS A140285 intersection target.
- `consecutive-digits`: ascending/descending cyclic consecutive decimal digits,
  OEIS A118569 intersection target.

The membership test tracks the candidate's survivor rank:

1. Start at `rank = (n + 1) / 2`, the rank among odd numbers after deleting evens.
2. For each lucky deletion factor `l`, reject when `rank % l == 0`.
3. Otherwise update `rank -= rank / l`.
4. Once `rank < l`, the candidate has survived all future deletions and is lucky.

The deletion factors are generated by a finite lucky sieve over odd values. The
sieve uses a packed `u64` bitset for alive odd indices and a Fenwick tree over
word popcounts to support `select(rank)` and deletion updates. No skip list is
used.

Dense early deletion passes use a faster path: the first `factor = 3` deletion
is a parallel periodic mask over the bitset, and later dense small-factor passes
scan disjoint word ranges in parallel using a precomputed 16-bit transition
table. After each scan pass, the Fenwick tree is rebuilt. Sparse later passes
still use `select(rank)` plus point updates, since scanning the whole bitset is
then slower than deleting a small number of survivors.

## Usage

```sh
cargo run --release -- --self-test
cargo run --release -- --memory-mib 512 --progress-seconds 2
cargo run --release -- --family all --memory-mib 8192 --progress-seconds 30
cargo run --release -- --family repdigit,mersenne,fibonacci --limit 10000000000 --memory-mib 2048
```

By default the search starts at `0`, so it prints the small repdigits first and
then continues through odd repdigits up to `u64::MAX`. Without `--memory-mib`, it
sizes the sieve from `/proc/meminfo` `MemAvailable` minus a 1 GiB reserve. Use
`--limit` to cap the largest generated deletion factor.

```sh
cargo run --release -- --min 4294967296 --memory-mib 8192
cargo run --release -- --family all --min 1000000000 --memory-mib 8192
```

Set `--scan-threshold 0` to disable the scan/rebuild deletion path for
comparison. The default scan threshold is `511`; values above `2048` are ignored
by the packed transition-table implementation. Small sieves use a conservative
dense-pass cutoff; multi-GB sieves scan somewhat deeper to avoid very large
Fenwick point-update passes.
